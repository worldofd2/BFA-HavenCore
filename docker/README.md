# BFA-HavenCore in Docker (Linux, experimental) — step-by-step guide

**Languages / Idiomas:** [English](#english) | [Español](#español)

---

## English

The main project README only documents building on Windows with Visual
Studio 2022. This setup ports the build to Linux (Ubuntu 24.04 + GCC 13) so
you can build and run `worldserver` and `bnetserver` with `docker compose`,
without installing anything on the host besides Docker.

**Status:** validated end-to-end with real data — `docker compose build`
compiles everything, `bnetserver` and `worldserver` start, connect to MySQL,
load DBC/maps/vmaps, and `worldserver` reaches `World initialized ...
ready`, with no restarts. Only one source-code change was needed to port it
to Linux (`src/server/worldserver/RemoteAccess/RASession.cpp`, missing
`<chrono>`/`<thread>` includes, typical of code that was only ever compiled
with MSVC) plus two config tweaks that `entrypoint.sh` applies automatically
(see [step 6](#6-start-bnetserver-and-worldserver) and
[troubleshooting](#10-troubleshooting)). The map-extraction tools (`TOOLS`)
remain **disabled** in the build — you don't need them if you already have
`./client-data` generated some other way (see
[step 7](#7-client-data-mapsvmapsmmapsdbc)).

#### 1. Prerequisites

- **Docker Desktop** installed and running, in Linux containers mode (the
  default; Windows containers are not needed).
- ~10 GB of free disk space (images + MySQL volumes).
- Nothing else — Boost, MySQL, OpenSSL, the compiler, etc. all live inside
  the Docker images. No need to install anything on the host.

Check that Docker responds:

```bash
docker --version
docker compose version
```

#### 2. Getting what's missing (not included in the repo)

This repo **does not include** two things you need for a playable server:

1. **The base SQL dumps** (`bfa_auth`, `bfa_world`, `bfa_characters`,
   `bfa_hotfixes`). The `sql/base/` folder is empty and gitignored on
   purpose. Get them from the project's Discord (see the main README).
   Once downloaded, place the `.sql` files in:

   ```
   BFA-HavenCore/sql/base/*.sql
   ```

2. **The extracted client data** (`dbc`, `maps`, `vmaps`, `mmaps`), needed
   for `worldserver` to load the world. These are generated by running the
   extraction tools against a copy of the WoW 8.3.7 client. You can bring
   the stack up without this just to check that it builds and starts, but
   the world won't finish loading — see
   [step 7](#7-client-data-mapsvmapsmmapsdbc).

If you don't have either of these yet, you can still follow steps 3-6 to
validate that the build and startup work — you'll see `worldserver`/
`bnetserver` connect to MySQL fine and fail in a controlled way due to
missing tables. That's expected.

#### 3. Building the images

From the repo root:

```bash
docker compose build
```

The first time takes **~10-15 minutes** (it compiles the entire WoW core
from scratch, ~1900 C++ files). Later runs reuse Docker's cached layers and
only recompile if the source changed.

When it's done you should have two new images:

```bash
docker images | grep bfa-havencore
# bfa-havencore-bnetserver    latest    ...
# bfa-havencore-worldserver   latest    ...
```

#### 4. Starting MySQL

```bash
docker compose up -d mysql
```

Wait for the healthcheck to pass (a few seconds):

```bash
docker compose ps
# bfacore-mysql   ...   Up ... (healthy)   0.0.0.0:3306->3306/tcp
```

This already auto-creates the 4 empty databases (`bfa_auth`, `bfa_world`,
`bfa_characters`, `bfa_hotfixes`) via `docker/mysql-init/`.

#### 5. Importing the databases

With the `.sql` files from [step 2](#2-getting-whats-missing-not-included-in-the-repo)
already copied into `./sql/base/`:

```bash
docker compose run --rm db-import
```

This walks `./sql/base/*.sql` and imports them one by one into MySQL (the
real dumps weigh several hundred MB — `bfa_world.sql` and
`bfa_hotfixes.sql` in particular can take a few minutes). If the folder is
empty, the command warns and does nothing.

> `entrypoint.sh` forces `Updates.EnableDatabases = 0` in the generated
> `.conf` files. TrinityCore's auto-updater (`DBUpdater`) needs the `mysql`
> binary (not present in the runtime image) and the `sql/` tree exactly as
> it existed at `/src` during compilation (doesn't exist at runtime) —
> since the schema here is managed by importing the dump by hand, it's
> disabled to avoid the server exiting with `exit 1` without logging why.
> See [troubleshooting](#10-troubleshooting) if you'd rather enable it.

#### 6. Start bnetserver and worldserver

```bash
docker compose up -d bnetserver worldserver
docker compose logs -f worldserver bnetserver
```

The first time each container starts, `entrypoint.sh` generates its config
file (`worldserver.conf` / `bnetserver.conf`) from the project's
`.conf.dist`: it points the database connection at the `mysql` service,
disables the schema auto-updater (see previous step), and fixes the
Windows-style path separators that ship in the `.dist` files
(`DataDir = ".\ClientData"` → `"./ClientData"`; on Linux a backslash is
just another character in a filename, not a path separator, so without
this fix `worldserver` can't find the client data). Those `.conf` files are
kept in Docker volumes (`worldserver_etc`, `bnetserver_etc`), so they
survive restarts and can be edited by hand without being overwritten on the
next start (see [step 9](#9-everyday-useful-commands)).

With the databases imported and `./client-data` populated, `worldserver`'s
logs will show something like this (it takes **~1-2 minutes**, which is
expected):

```
INFO  Using DataDir ./ClientData/
INFO  Initialize data stores...
...
INFO  World initialized in 1 minutes 26 seconds
INFO  BfaCore rev. ... ready...
```

If the databases are NOT imported, you'll see the containers restart in a
loop with errors like `Table 'bfa_auth.account' doesn't exist` — go back to
[step 5](#5-importing-the-databases).

##### Exposed ports

| Service     | Port | Use                                                    |
|-------------|------|----------------------------------------------------------|
| mysql       | 3306 | MySQL                                                    |
| bnetserver  | 1119 | Battle.net protocol                                      |
| bnetserver  | 8081 | Login REST (launcher)                                    |
| worldserver | 8085 | WorldServerPort — initial connection / character select |
| worldserver | 8086 | InstanceServerPort — hand-off when clicking "Enter World"|

> **`worldserver` needs two ports, not one.** When you click "Enter World",
> the server sends the client a `ConnectTo` packet (see
> `WorldSession::SendConnectToInstance` in
> `src/server/game/Server/WorldSession.cpp`) telling it to open a *second*
> connection to `InstanceServerPort` (default **8086**, defined in
> `World.cpp`, not in the `.conf`). If you only publish 8085, login and
> character select work fine (they use that connection), but when entering
> the world the second connect never reaches the container and the client
> shows **"World server is down"**. `docker-compose.yml` already publishes
> both.

#### 7. Client data: maps/vmaps/mmaps/dbc

`worldserver` looks for this data in `./client-data` (mounted as
`/opt/bfacore/bin/ClientData`, read-only), with this layout:

```
client-data/
├── cameras/
├── dbc/
├── gt/
├── maps/
├── vmaps/
└── mmaps/       (optional — creature pathing is degraded without it)
```

These are generated with `map_extractor`, `vmap4_extractor`,
`vmap4_assembler` and `mmaps_generator` run against the WoW 8.3.7 client
files. These tools are **disabled** in the Docker build (`-DTOOLS=0` in
`docker/Dockerfile`) because compiling them on Linux hasn't been validated
— the tested path is generating them elsewhere (for example with the
Windows build from the main README) and copying the result here. If you'd
rather generate them on Linux:

1. In `docker/Dockerfile`, change `-DTOOLS=0` to `-DTOOLS=1`.
2. `docker compose build` again — you'll probably hit more missing includes
   (the same kind of fix as `RASession.cpp`); fix them one by one like the
   first time.
3. Run the extractors against a folder with the game client mounted as a
   volume, and copy the result into `./client-data`.

Without `./client-data` populated you can still bring the stack up to test
startup, the database connection, and account creation — the world itself
just won't finish loading.

#### 8. Creating an account and connecting the game client

`worldserver` has an interactive console. Since the service is defined with
`stdin_open`/`tty` in `docker-compose.yml`, you can attach to it like this:

```bash
docker attach bfacore-worldserver
```

And inside the console:

```
account create myusername mypassword
account set gmlevel myusername 3 -1
```

(`Ctrl+P` then `Ctrl+Q` to detach without killing the process.)

> **Note:** this fork uses Battle.net-style login (linked to `bnetserver`).
> `account create` alone may not be enough — see the
> [troubleshooting](#10-troubleshooting) note below on `bnetaccount create`,
> which is the command that actually worked in testing.

To connect the game client, point the `realmlist` at the IP/host where
Docker is running, port `8085` (worldserver), and the launcher/login at
port `8081` (bnetserver, Login REST). The exact `realmlist`/`realm` details
in the `bfa_auth.realmlist` table depend on the dump you imported — you may
need to update that row with the host's real IP.

#### 9. Everyday useful commands

```bash
# Check the status of all services
docker compose ps

# Live logs for a service
docker compose logs -f worldserver

# Restart a service (e.g. after editing its .conf)
docker compose restart worldserver

# Get a shell in a running container
docker compose exec worldserver bash

# Edit the already-generated .conf (lives in a volume, not in the repo)
docker compose exec worldserver sh -c "vi /opt/bfacore/etc/worldserver.conf"

# Stop everything (keeps the volumes: MySQL data and the .conf files)
docker compose down

# Stop everything and also remove the volumes (start from scratch)
docker compose down -v

# Rebuild after touching source code
docker compose build worldserver bnetserver
docker compose up -d worldserver bnetserver
```

#### 10. Troubleshooting

**`bnetserver`/`worldserver` keep restarting in a loop right after
starting.**
Check the logs (`docker compose logs worldserver`). If it says `Table ...
doesn't exist`, you missed [step 5](#5-importing-the-databases). If it says
something about a refused connection to MySQL, wait for `mysql` to be
`healthy` (`docker compose ps`).

**`docker compose build` fails with a compile error (`error: ...`).**
Can happen if you touched source code. The most common pattern seen during
the port was missing standard-library includes (`<thread>`, `<chrono>`,
etc.) because the original code was only ever compiled with MSVC, which
includes them transitively. Add the missing include to the file the error
points at and rebuild.

**`worldserver` starts, prints the banner + Boost version, and exits with
`exit 1` with no visible error in the log.**
This is the symptom of the schema auto-updater (`DBUpdater`) not finding
the `mysql` binary or the compile-time `sql/` tree inside the runtime image
— the failure log (`TC_LOG_FATAL`) gets lost and the process exits before
flushing it. `entrypoint.sh` already prevents this by setting
`Updates.EnableDatabases = 0`, but if you edited the `.conf` by hand and it
happened again, check that line. If instead you *want* the auto-updater to
run (for example to apply `sql/updates/*` in the future), you'll need to:
install `default-mysql-client` in the `runtime` stage of the `Dockerfile`,
and mount the repo (or at least `./sql`) inside the container at the same
path that `/src` had during the build.

**`worldserver` starts but can't find `ClientData` / maps even though it's
mounted.**
Check that `DataDir`/`LogsDir` in the generated `.conf` use `/` and not `\`
(`docker compose exec worldserver grep DataDir /opt/bfacore/etc/worldserver.conf`).
`entrypoint.sh` fixes this automatically the first time it generates the
`.conf`; if the `.conf` already existed from an old run (a `worldserver_etc`
volume from before this fix), delete it and let it regenerate:
`docker compose down` followed by
`docker volume rm bfa-havencore_worldserver_etc`.

**You can log in, reach character select, even create a character — but
clicking "Enter World" shows "World server is down".**
You're missing port `8086` (`InstanceServerPort`). The initial login and
character select use `WorldServerPort` (8085), but when entering the world
the server sends the client a `ConnectTo` asking it to open a *second*
connection to `InstanceServerPort` (8086 by default — it's hardcoded in
`World.cpp`, it doesn't appear in the `.conf`). If that port isn't
published in `docker-compose.yml`, that second connection never reaches the
container and the client reads it as the server being down. It's already
added in this repo's compose file (`"8086:8086"` on the `worldserver`
service) — if you removed it or you're on an old copy, add it and run
`docker compose up -d --force-recreate worldserver`.

**Creating the first account: `account create` says "There is no such
subcommand".**
This fork doesn't have a plain `account create` — it uses Battle.net-style
login. Use `bnetaccount create <email> <password>` instead (the account
name must contain an `@`). It automatically creates both the Battle.net
account and a linked game account (e.g. `3#1`), which is the name you then
use with `account set gmlevel <gameaccount> <level> -1`.

**I want to change the MySQL root password.**
Create a `.env` file at the repo root with `DB_ROOT_PASSWORD=your_password`
before starting `mysql` for the first time (if the MySQL volume already
exists with a different password, remove it first with
`docker compose down -v`).

**I want to see how big the images are.**
```bash
docker images | grep bfa-havencore
```

#### 11. Notes / known limitations

- Built with `SCRIPTS=static`, `RelWithDebInfo`, with precompiled headers
  enabled (same as on Windows).
- The runtime image runs as root for simplicity; if this goes to
  production, it's worth adding an unprivileged user in the Dockerfile.
- TrinityCore's schema auto-updater (`DBUpdater`, applies `sql/updates/*`
  automatically at startup) is disabled — here the schema is managed by
  importing the base dump by hand. This repo also doesn't ship
  `sql/updates/` yet, so nothing is lost for now.
- `mmaps` is optional: if it's missing from `./client-data`, the server
  still starts, but creature pathing may be worse.
- Validated with a real set of SQL dumps (`bfa_auth`, `bfa_characters`,
  `bfa_hotfixes`, `bfa_world`) and full client data (`dbc`/`maps`/`vmaps`,
  without `mmaps`): clean build, `worldserver` reaches
  `World initialized ... ready` in ~1-2 minutes, `bnetserver` up with no
  restarts, and a real client login all the way into the world (character
  select, character creation, and "Enter World") was confirmed working
  once port 8086 was published.

#### 1. Requisitos previos

- **Docker Desktop** instalado y corriendo, en modo contenedores Linux (es
  el default; no hace falta Windows containers).
- ~10 GB libres en disco (imágenes + volúmenes de MySQL).
- Nada más — Boost, MySQL, OpenSSL, el compilador, etc. viven todos dentro
  de las imágenes Docker. No hace falta instalar nada en el host.

Verificá que Docker responde:

```bash
docker --version
docker compose version
```

#### 2. Conseguir lo que falta (no viene en el repo)

Este repo **no incluye** dos cosas necesarias para tener un server jugable:

1. **Los dumps SQL base** (`bfa_auth`, `bfa_world`, `bfa_characters`,
   `bfa_hotfixes`). La carpeta `sql/base/` está vacía y gitignoreada a
   propósito. Se consiguen por el Discord del proyecto
   (ver README principal). Una vez descargados, colocá los `.sql` en:

   ```
   BFA-HavenCore/sql/base/*.sql
   ```

2. **Los datos extraídos del cliente** (`dbc`, `maps`, `vmaps`, `mmaps`),
   necesarios para que `worldserver` cargue el mundo. Se generan corriendo
   las herramientas de extracción contra una copia del cliente de WoW
   8.3.7. Podés levantar el stack sin esto para probar que compila y
   arranca, pero el mundo no va a terminar de cargar — ver
   [paso 7](#7-datos-del-cliente-mapasvmapsmmapsdbc).

Si todavía no tenés ninguna de las dos cosas, podés seguir igual los pasos
3 a 6 para validar que el build y el arranque funcionan — vas a ver a
`worldserver`/`bnetserver` conectarse bien a MySQL y fallar de forma
controlada por falta de tablas. Es el comportamiento esperado.

#### 3. Construir las imágenes

Desde la raíz del repo:

```bash
docker compose build
```

La primera vez tarda **~10-15 minutos** (se compila todo el core de WoW
desde cero, ~1900 archivos C++). Las siguientes veces Docker reutiliza las
capas cacheadas y solo recompila si cambió el código fuente.

Al terminar deberías tener dos imágenes nuevas:

```bash
docker images | grep bfa-havencore
# bfa-havencore-bnetserver    latest    ...
# bfa-havencore-worldserver   latest    ...
```

#### 4. Levantar MySQL

```bash
docker compose up -d mysql
```

Esperá a que el healthcheck pase (unos segundos):

```bash
docker compose ps
# bfacore-mysql   ...   Up ... (healthy)   0.0.0.0:3306->3306/tcp
```

Esto ya crea automáticamente las 4 bases vacías (`bfa_auth`, `bfa_world`,
`bfa_characters`, `bfa_hotfixes`) vía `docker/mysql-init/`.

#### 5. Importar las bases de datos

Con los `.sql` del [paso 2](#2-conseguir-lo-que-falta-no-viene-en-el-repo)
ya copiados en `./sql/base/`:

```bash
docker compose run --rm db-import
```

Esto recorre `./sql/base/*.sql` y los importa uno por uno a MySQL (los dumps
reales pesan varios cientos de MB — `bfa_world.sql` y `bfa_hotfixes.sql` en
particular pueden tardar unos minutos). Si la carpeta está vacía, el comando
avisa y no hace nada.

> El `entrypoint.sh` fuerza `Updates.EnableDatabases = 0` en los `.conf`
> generados. El auto-updater de TrinityCore (`DBUpdater`) necesita el
> binario `mysql` (no está en la imagen runtime) y el árbol `sql/` tal como
> existía en `/src` durante la compilación (no existe en runtime) — como
> acá el esquema se maneja importando el dump a mano, se desactiva para
> evitar que el server salga con `exit 1` sin loguear el motivo. Ver
> [troubleshooting](#10-troubleshooting-1) si preferís habilitarlo.

#### 6. Levantar bnetserver y worldserver

```bash
docker compose up -d bnetserver worldserver
docker compose logs -f worldserver bnetserver
```

La primera vez que arranca cada contenedor, el `entrypoint.sh` genera su
archivo de config (`worldserver.conf` / `bnetserver.conf`) a partir del
`.conf.dist` del proyecto: apunta la conexión de base de datos al servicio
`mysql`, desactiva el auto-updater de esquema (ver paso anterior) y corrige
los paths con separador de Windows que traen los `.dist`
(`DataDir = ".\ClientData"` → `"./ClientData"`; en Linux una barra invertida
es un carácter más del nombre de archivo, no un separador de ruta, así que
sin este fix `worldserver` no encuentra los datos del cliente). Esos `.conf`
quedan guardados en volúmenes Docker (`worldserver_etc`, `bnetserver_etc`),
así que sobreviven a reinicios y se pueden editar a mano sin que se pisen en
el próximo arranque (ver [paso 9](#9-comandos-útiles-del-día-a-día)).

Con las bases importadas y `./client-data` poblado, en los logs de
`worldserver` vas a ver algo como (tarda **~1-2 minutos**, es esperable):

```
INFO  Using DataDir ./ClientData/
INFO  Initialize data stores...
...
INFO  World initialized in 1 minutes 26 seconds
INFO  BfaCore rev. ... ready...
```

Si las bases NO están importadas, vas a ver los contenedores reiniciándose
en loop con errores tipo `Table 'bfa_auth.account' doesn't exist` — volvé
al [paso 5](#5-importar-las-bases-de-datos).

##### Puertos expuestos

| Servicio    | Puerto | Uso                                                    |
|-------------|--------|---------------------------------------------------------|
| mysql       | 3306   | MySQL                                                   |
| bnetserver  | 1119   | Protocolo Battle.net                                    |
| bnetserver  | 8081   | Login REST (launcher)                                   |
| worldserver | 8085   | WorldServerPort — conexión inicial / character select   |
| worldserver | 8086   | InstanceServerPort — hand-off al hacer "Enter World"     |

> **`worldserver` necesita dos puertos, no uno.** Al hacer clic en "Enter
> World", el servidor manda al cliente un paquete `ConnectTo` (ver
> `WorldSession::SendConnectToInstance` en
> `src/server/game/Server/WorldSession.cpp`) diciéndole que abra una
> *segunda* conexión a `InstanceServerPort` (por defecto **8086**, definido
> en `World.cpp`, no en el `.conf`). Si solo publicás el 8085, el login y
> el character select andan perfecto (usan esa conexión), pero al entrar al
> mundo el segundo connect nunca llega al contenedor y el cliente muestra
> **"World server is down"**. `docker-compose.yml` ya publica ambos.

#### 7. Datos del cliente: mapas/vmaps/mmaps/dbc

`worldserver` busca esos datos en `./client-data` (montado como
`/opt/bfacore/bin/ClientData`, solo lectura), con esta estructura:

```
client-data/
├── cameras/
├── dbc/
├── gt/
├── maps/
├── vmaps/
└── mmaps/       (opcional — pathing de creaturas degradado sin esto)
```

Se generan con `map_extractor`, `vmap4_extractor`, `vmap4_assembler` y
`mmaps_generator` corriendo contra los archivos del cliente de WoW 8.3.7.
Estas herramientas están **desactivadas** en el build de Docker
(`-DTOOLS=0` en `docker/Dockerfile`) porque no se validó que compilen en
Linux — la forma probada es generarlas aparte (por ejemplo con el build de
Windows del README principal) y copiar el resultado acá. Si preferís
generarlas dentro de Linux:

1. En `docker/Dockerfile`, cambiar `-DTOOLS=0` por `-DTOOLS=1`.
2. `docker compose build` de nuevo — probablemente aparezcan más includes
   faltantes (mismo tipo de fix que en `RASession.cpp`); hay que corregirlos
   uno por uno igual que se hizo la primera vez.
3. Correr los extractores contra una carpeta con el cliente del juego
   montada como volumen, y copiar el resultado a `./client-data`.

Sin `./client-data` poblado podés igual levantar el stack para probar el
arranque, la conexión a la base y la creación de cuentas — el mundo en sí
no va a terminar de cargar.

#### 8. Crear una cuenta y conectar el cliente del juego

`worldserver` tiene una consola interactiva. Como el servicio está
definido con `stdin_open`/`tty` en el `docker-compose.yml`, te podés
enganchar así:

```bash
docker attach bfacore-worldserver
```

Y dentro de la consola:

```
account create miusuario mipassword

```

(`Ctrl+P` seguido de `Ctrl+Q` para desengancharte sin matar el proceso.)

> **Nota:** este fork usa login estilo Battle.net (vinculado a
> `bnetserver`). Puede que `account create` solo no alcance — ver la nota de
> [troubleshooting](#10-troubleshooting-1) sobre `bnetaccount create`, que
> fue el comando que realmente funcionó en las pruebas.

Para conectar el cliente del juego, apuntá el `realmlist` a la IP/host
donde corre Docker, puerto `8085` (worldserver), y el launcher/login al
puerto `8081` (bnetserver, Login REST). Los detalles exactos de
`realmlist`/`realm` en la tabla `bfa_auth.realmlist` dependen del dump que
hayas importado — puede que necesites actualizar esa fila con la IP real
del host.

#### 9. Comandos útiles del día a día

```bash
# Ver estado de todos los servicios
docker compose ps

# Logs en vivo de un servicio
docker compose logs -f worldserver

# Reiniciar un servicio (por ejemplo tras editar su .conf)
docker compose restart worldserver

# Entrar a un contenedor corriendo
docker compose exec worldserver bash

# Editar el .conf ya generado (vive en un volumen, no en el repo)
docker compose exec worldserver sh -c "vi /opt/bfacore/etc/worldserver.conf"

# Parar todo (mantiene los volúmenes: datos de MySQL y los .conf)
docker compose down

# Parar todo y borrar también los volúmenes (arranca de cero)
docker compose down -v

# Recompilar después de tocar código fuente
docker compose build worldserver bnetserver
docker compose up -d worldserver bnetserver
```

#### 10. Troubleshooting

**`bnetserver`/`worldserver` reinician en loop apenas arrancan.**
Mirá los logs (`docker compose logs worldserver`). Si dice `Table ... doesn't
exist`, faltó el [paso 5](#5-importar-las-bases-de-datos). Si dice algo de
conexión rechazada a MySQL, esperá a que `mysql` esté `healthy`
(`docker compose ps`).

**`docker compose build` falla con un error de compilación (`error: ...`).**
Puede pasar si tocaste código fuente. El patrón más común visto durante el
port fue includes de la librería estándar que faltan (`<thread>`,
`<chrono>`, etc.) porque el código original solo se compiló con MSVC, que
los incluye transitivamente. Agregá el include que falte al archivo que
señala el error y recompilá.

**`worldserver` arranca, imprime el banner + versión de Boost, y sale con
`exit 1` sin ningún error visible en el log.**
Es el síntoma que dio el auto-updater de esquema (`DBUpdater`) al no
encontrar el binario `mysql` ni el árbol `sql/` de compilación dentro de la
imagen runtime — el log del fallo (`TC_LOG_FATAL`) se pierde y el proceso
corta antes de flushearlo. El `entrypoint.sh` ya lo evita seteando
`Updates.EnableDatabases = 0`, pero si editaste el `.conf` a mano y volvió a
pasar, revisá esa línea. Si en cambio *querés* que el auto-updater corra
(por ejemplo para aplicar `sql/updates/*` en el futuro), vas a necesitar:
instalar `default-mysql-client` en la etapa `runtime` del `Dockerfile`, y
montar el repo (o al menos `./sql`) dentro del contenedor en la misma ruta
que tenía `/src` durante el build.

**`worldserver` arranca pero no encuentra `ClientData` / mapas pese a estar
montado.**
Revisá que `DataDir`/`LogsDir` en el `.conf` generado usen `/` y no `\`
(`docker compose exec worldserver grep DataDir /opt/bfacore/etc/worldserver.conf`).
El `entrypoint.sh` corrige esto automáticamente la primera vez que genera el
`.conf`; si el `.conf` ya existía de una corrida vieja (volumen
`worldserver_etc` de antes de este fix), borralo y dejá que se regenere:
`docker compose down` seguido de `docker volume rm bfa-havencore_worldserver_etc`.

**Loguea, entra a character select, hasta puede crear personaje — pero al
apretar "Enter World" el cliente muestra "World server is down".**
Te falta publicar el puerto `8086` (`InstanceServerPort`). El login inicial
y el character select usan `WorldServerPort` (8085), pero al entrar al
mundo el servidor le manda al cliente un `ConnectTo` pidiéndole que abra
una *segunda* conexión a `InstanceServerPort` (8086 por defecto — está
hardcodeado en `World.cpp`, no aparece en el `.conf`). Si ese puerto no
está publicado en `docker-compose.yml`, esa segunda conexión nunca llega al
contenedor y el cliente lo interpreta como servidor caído. Ya está
agregado en el compose de este repo (`"8086:8086"` en el servicio
`worldserver`) — si lo sacaste o estás en una copia vieja, agregalo y
`docker compose up -d --force-recreate worldserver`.

**Al crear la primera cuenta: `account create` dice "There is no such
subcommand".**
Este fork no tiene un `account create` a secas — usa login estilo
Battle.net. Usá `bnetaccount create <email> <password>` en su lugar (el
nombre de cuenta tiene que tener un `@`). Esto crea automáticamente la
cuenta Battle.net y una cuenta de juego vinculada (por ejemplo `3#1`), que
es el nombre que después usás con
`account set gmlevel <cuentadejuego> <nivel> -1`.

**Quiero cambiar la contraseña de root de MySQL.**
Creá un archivo `.env` en la raíz del repo con `DB_ROOT_PASSWORD=tu_clave`
antes de levantar `mysql` por primera vez (si ya existe el volumen de
MySQL con otra clave, borralo primero con `docker compose down -v`).

**Quiero ver cuánto pesan las imágenes.**
```bash
docker images | grep bfa-havencore
```

#### 11. Notas / limitaciones conocidas

- Build en modo `SCRIPTS=static`, `RelWithDebInfo`, con precompiled headers
  activados (igual que en Windows).
- La imagen runtime corre como root por simplicidad; si esto va a
  producción, conviene agregar un usuario sin privilegios en el
  Dockerfile.
- El auto-updater de esquema de TrinityCore (`DBUpdater`, aplica
  `sql/updates/*` automáticamente al arrancar) queda desactivado — acá el
  esquema se maneja importando el dump base a mano. Este repo tampoco trae
  `sql/updates/` todavía, así que no se pierde nada por ahora.
- `mmaps` es opcional: si no está en `./client-data`, el server arranca
  igual pero el pathing de creaturas puede ser peor.
- Validado con un set real de dumps SQL (`bfa_auth`, `bfa_characters`,
  `bfa_hotfixes`, `bfa_world`) y datos de cliente completos
  (`dbc`/`maps`/`vmaps`, sin `mmaps`): build limpio, `worldserver` llega a
  `World initialized ... ready` en ~1-2 minutos, `bnetserver` arriba sin
  reinicios, y se confirmó un login real desde el cliente del juego hasta
  entrar al mundo (character select, creación de personaje y "Enter
  World") una vez publicado el puerto 8086.
